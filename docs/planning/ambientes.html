Ambientes

Os três ambientes da plataforma — o que cada um é, o que pode e o que não pode — em linguagem simples.

Isto é a especificação dos ambientes (jul/2026), não a criação deles na nuvem. Hospedagem (Vercel), domínios e dados de teste vêm a seguir. Detalhe técnico: docs/planning/templates/environments.md · decisões D-025 e D-026.

Em uma frase

Trabalhamos em três ambientes: o computador do desenvolvedor (local), um espaço privado de validação (staging) e o sistema real dos clientes (produção). Em teste nunca se abre uma porta real nem se manda mensagem paga a um cliente real.

Os três ambientes

AmbientePara quêDadosIntegraçõesQuem usa

staging Validar antes de ir ao ar Dados de teste / anonimizados — sem copiar a base de produção por padrão Alvos seguros de teste (sandbox, dispositivo de teste) Desenvolvimento + UAT opcional dos sócios (privado)
produção Sistema ao vivo Dados reais, protegidos pela LGPD Serviços reais (fechadura, WhatsApp, CRM, etc.) Clientes, corretores, equipe

Regras que importam

  • Staging antes de produção — o caminho normal é validar no staging e só então promover. Não se sobe direto do computador para produção.
  • Staging não é beta público — não é um ambiente para corretores ou clientes em geral; é privado.
  • Local isolado — no dia a dia o site corre no computador com Node (Nuxt), com base e integrações locais/simuladas. Não é “rodar a Vercel” na máquina. Ligar o local ao staging só de propósito; nunca à produção.
  • Como a aplicação sabe onde está — uma variável chamada APP_ENV com valor local, staging ou prod.
  • Pré-visualizações na hospedagem (Preview) — URL temporária de um feat/… ou fix/…; usam a mesma classe de backends do staging — não são um quarto ambiente.

Branch feat/… ou fix/… (Preview) staging Rascunho na nuvem, ainda em modo teste
Branch staging (permanente) staging Validação compartilhada / UAT dos sócios (privado)
Branch main produção Sistema ao vivo

Caminho normal: trabalho numa feature → entra em staging → só depois se promove para main (produção). A Vercel não promove sozinha: alguém (ou um passo de release) decide quando staging vira produção.

Fechar um change (no fluxo de desenvolvimento) deve aterrar em staging. Nota: a ferramenta automática de close ainda faz merge para main até atualizarmos o skill com um arquivo de config opcional — isso fica para uma tarefa seguinte (#166).

Correção urgente (hotfix)

Em emergência pode ser preciso corrigir produção fora do caminho normal. Isso existe no desenho como exceção: tem de ser explícita, excepcional e registrada. O passo a passo detalhado será definido quando desenharmos a promoção e o release — não inventamos o procedimento aqui.
