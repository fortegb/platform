Ambientes

Os três ambientes da plataforma — o que cada um é, o que pode e o que não pode — em linguagem simples.

Isto é a especificação dos ambientes (jul/2026), não a criação deles na nuvem. Detalhe técnico: docs/planning/templates/environments.md · decisões D-025 a D-036.

Em uma frase

Trabalhamos em três ambientes: o computador do desenvolvedor (local), um espaço privado de validação (staging) e o sistema real dos clientes (produção). Em teste nunca se abre uma porta real nem se manda mensagem paga a um cliente real.

Os três ambientes

AmbientePara quêDadosIntegraçõesQuem usa
local Desenvolver e depurar no computador Dados fictícios / de teste — sem dados reais de clientes Simulações (mocks) — sem fechadura nem WhatsApp reais Quem desenvolve
staging Validar antes de ir ao ar Dados de teste / anonimizados — sem copiar a base de produção por padrão Alvos seguros de teste (sandbox, dispositivo de teste) Desenvolvimento + UAT opcional dos sócios (privado)
produção Sistema ao vivo Dados reais, protegidos pela LGPD Serviços reais (fechadura, WhatsApp, CRM, etc.) Clientes, corretores, equipe

Regras que importam

  • Staging antes de produção — o caminho normal é validar no staging e só então promover. Não se sobe direto do computador para produção.
  • Staging não é beta público — não é um ambiente para corretores ou clientes em geral; é privado.
  • Local isolado — no dia a dia o site corre no computador com Node (Nuxt), com base e integrações locais/simuladas. Não é “rodar a Vercel” na máquina. Ligar o local ao staging só de propósito; nunca à produção.
  • Como a aplicação sabe onde está — uma variável chamada APP_ENV com valor local, staging ou prod.
  • Pré-visualizações na hospedagem (Preview) — URL temporária de um feat/… ou fix/…; usam a mesma classe de backends do staging — não são um quarto ambiente.

Branches (qual linha git é qual ambiente)

Onde está o códigoAmbienteEm linguagem simples
No seu computador local Desenvolvimento diário (npm run dev)
Branch feat/… ou fix/… (Preview) staging Rascunho na nuvem, ainda em modo teste
Branch staging (permanente) staging Validação compartilhada / UAT dos sócios (privado)
Branch main produção Sistema ao vivo

Caminho normal: trabalho numa feature → entra em staging → só depois se promove para main (produção). A Vercel não promove sozinha: alguém (ou um passo de release) decide quando staging vira produção.

Fechar um change (no fluxo de desenvolvimento) deve aterrar em staging. Nota: a ferramenta automática de close ainda faz merge para main até atualizarmos o skill com um arquivo de config opcional — isso fica para uma tarefa seguinte (#166).

Hospedagem (Vercel)

Usamos um único projeto na Vercel:

  • Produção — só o que está em main (site ao vivo).
  • Preview — staging e branches de feature: URLs de teste. Existem ao mesmo tempo que a produção; não é preciso “desligar” o site real para testar. Se ninguém acessa há um tempo, o primeiro clique pode demorar um pouco (cold start).

Acesso ao staging / Previews: uma senha compartilhada na porta da Vercel (sócios não precisam de conta Vercel). Digita uma vez no navegador → o site de teste inteiro fica liberado (até expirar o cookie ou limpar dados do browser). O login da plataforma (quando existir na página) é outra camada, depois dessa senha.

Dados (Supabase)

A base de dados, login e ficheiros sensíveis ficam no Supabase. No plano gratuito cabem dois projetos activos na nuvem — usamos exactamente isso, mais um ambiente local no computador:

Onde corre o appSupabaseEm linguagem simples
local No computador (Docker) Não gasta um dos dois projetos grátis da nuvem
staging + Previews Projeto fortegb-staging Um só projeto de teste partilhado (incluindo branches de feature)
produção Projeto fortegb-prod Dados reais dos clientes — separado do teste
  • Sem dados reais de clientes no teste — staging e local usam dados fictícios / de seed.
  • Mesma estrutura de tabelas nos três sítios; o que muda são os dados e as chaves secretas.
  • Plano gratuito: se o staging ficar sem uso ~7 dias, a Supabase pode “pausar” o projeto — reativa-se no painel. Limites de espaço (BD/ficheiros) existem; quando apertar, avalia-se o plano Pro.
  • Criar os projetos e colar as chaves na Vercel é setup depois; aqui só está o contrato.
  • Estrutura das tabelas: ficheiros de migração no repositório (Supabase CLI) — iguais em local, staging e produção. Quem desenvolve aplica no Docker; na nuvem aplica-se primeiro no staging e só depois na produção. O deploy do site na Vercel não muda a base de dados sozinho.
  • No computador: sobe-se o Supabase local com Docker (no Mac preferimos OrbStack; Docker Desktop também serve). O passo a passo técnico está em docs/planning/templates/supabase-local.md. Criar a pasta supabase/ no repositório é setup/bootstrap depois — não faz parte só desta especificação.
  • Dados de teste (seed): o mesmo pacote sintético em local e staging — pessoas com aparência realista em português do Brasil (inventadas), documentos RG/CNH dummy só para teste, contas de login de corretor/staff documentadas. Pode apagar e recriar. Nunca se copia a base de produção. Detalhe: docs/planning/templates/seed-lgpd.md.
  • Conteúdo (Sanity): um projeto com dois datasets — staging (local, staging e Previews) e production (site ao vivo). Publicar conteúdo novo no ar exige um passo explícito de promoção (não acontece só porque o site fez deploy). Detalhe técnico em docs/planning/templates/environments.md.
  • Modelo de conteúdo: tipos house, blogPost, e stubs de timeline/media kit. Marketing (textos, fotos, preço de exibição) fica no Sanity; status da casa, fechadura e QR ficam no Supabase — o site junta os dois pelo mesmo ID. Detalhe: docs/planning/templates/cms-content-model.md.

Correção urgente (hotfix)

Em emergência pode ser preciso corrigir produção fora do caminho normal. Isso existe no desenho como exceção: tem de ser explícita, excepcional e registrada. O passo a passo detalhado será definido quando desenharmos a promoção e o release — não inventamos o procedimento aqui.
