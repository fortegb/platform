Infraestrutura: serverless vs persistente

As duas formas avaliadas para rodar a plataforma — e por que escolhemos serverless.

Avaliação de Grilling 0 (03/07/2026). Resumo para sócios/equipe em Arquitetura e decisões. Fonte técnica (canon): docs/planning/explore/runtime-serverless-vs-persistent.md · decisão D-017.

Contexto

A plataforma não é um site de conteúdo com poucas integrações — é um sistema pesado em integrações e orientado a eventos (bots WhatsApp/Telegram, sync HubSpot, dispositivos Tuya, trabalho agendado) e com um cliente mobile futuro provável (staff a aprovar pedidos com push). A escolha da forma de rodar precisa suportar isso.

Critérios (prioridade do Ricardo): free-first · zero-ops · simplicidade (Vercel) · escala baixa (centenas de usuários; 1–2 casas/ano) · foco no que é único.

Caso A — Serverless ESCOLHIDO

App Nuxt/Nitro em modo serverless na Vercel; rotas = funções; API-first (web = 1.º cliente; PWA/native/bots reutilizam). Sem processo sempre ligado.

Prós: grátis + zero-ops (nada sempre ligado); escala a zero; DX Vercel; sem servidor para operar.
Contras: lógica assíncrona espalhada por funções + agendador externo; cold starts; sem conexões persistentes; streaming ao vivo inadequado (mitigado: vídeo fica no fornecedor).

Caso B — Persistente (avaliado)

Um processo Node sempre ligado (Nuxt/Nitro em modo node-server). Como nunca dorme, o mesmo processo faz tudo: HTTP + API + webhooks, agendador in-process, websockets para real-time e pools de conexão quentes.

Bots / integraçõesTelegram (grátis) + WhatsApp; Tuya/HubSpot/Google com pools quentes⚠️ WA pago

Prós: async/bots/eventos/real-time naturais num só processo; menos serviços (pg-boss substitui QStash; websockets substituem push); sem cold starts; modelo familiar (servidor a correr).
Contras: grátis + sempre ligado + zero-ops não coexistem — Render grátis dorme (mata o agendador), Oracle é VM crua (operas OS/TLS/deploy), Fly/Railway são suaves mas ~US$5/mo; não escala a zero; mais DevOps para dev solo.

Decisão — Serverless (Vercel Hobby → Pro quando útil). As prioridades declaradas — free-first + zero-ops + simplicidade Vercel — apontam para serverless: nenhum host sempre-ligado oferece “não-dorme” e “sem-operação” de graça ao mesmo tempo. O que se abdica é a coerência do assíncrono (fica espalhado por funções + QStash), aceite pelo dono (“não me importo desde que funcione”). O Caso B seria escolhido se coerência/real-time pesassem mais que free-first/zero-ops — fica documentado para reavaliação.

Condições da escolha (olhos abertos)

  1. Adicionar QStash — é o que torna o assíncrono espalhado fiável (jobs com atraso + retries); o cron da Vercel sozinho é grosseiro.
  2. Não passar vídeo ao vivo pelo backend — câmaras ao vivo ficam na app/cloud do fornecedor; embed/link.
  3. Vercel Hobby agora → Pro quando útil — a 1–2 casas/ano o uso não dispara upgrade; Pro é escolha discricionária.
