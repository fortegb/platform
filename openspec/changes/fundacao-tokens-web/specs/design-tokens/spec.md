## ADDED Requirements

### Requirement: Documented token foundation
The system SHALL provide a single documented source of truth for brand colors, typography, spacing, and button styles, derived from the existing Home page dependency tree (`tailwind.config.js`, `assets/css/main.css`, and the components it renders) rather than inventing new values.

#### Scenario: Design leaf references tokens
- **WHEN** a Passo 6 design leaf (#197–#207) needs a color, font, spacing, or button style
- **THEN** it references the documented token inventory instead of re-deriving or hardcoding arbitrary values

### Requirement: No untokenized recurring brand color
A color value that repeats identically across multiple files for the same semantic role SHALL be wired into `tailwind.config.js` as a named token rather than left as a repeated arbitrary-value class.

#### Scenario: WhatsApp CTA color
- **WHEN** a component renders the WhatsApp call-to-action (button or link)
- **THEN** it uses the `whatsapp` / `whatsapp-hover` Tailwind color tokens instead of `bg-[#3E8E5E]` / `hover:bg-[#34784F]`

### Requirement: No dead or duplicate token-layer CSS
Custom CSS classes in the token layer (`main.css`) SHALL NOT sit unused, and SHALL NOT redeclare a class name already generated by the DaisyUI theme with the same effective value.

#### Scenario: Unused custom class
- **WHEN** a custom class in `main.css` has zero usages across the repo
- **THEN** it is removed rather than left as dead weight in the token layer

#### Scenario: Class name collision with generated utility
- **WHEN** a custom class in `main.css` shares its name with a class DaisyUI already generates from the active theme, and both resolve to the same value
- **THEN** the custom override is removed and the DaisyUI-generated class is relied on instead

### Requirement: Single component per structurally-identical hero layout
Hero layouts that share the same structure (grid, buttons, copy) and differ only in background gradient SHALL be implemented as one component accepting a `variant` prop, not as separate duplicated components.

#### Scenario: Gradient-only hero variant
- **WHEN** a page needs the split-grid hero with a specific gradient (`split`, `slate`, or `azul`)
- **THEN** it renders the consolidated `Hero` component with `variant` set accordingly, instead of a dedicated per-gradient component

#### Scenario: Structurally different hero stays separate
- **WHEN** a hero layout is structurally different (e.g. full-bleed, centered) rather than a gradient variant of the split layout
- **THEN** it remains its own component (`HeroClassic`) and is not folded into the `variant` prop
